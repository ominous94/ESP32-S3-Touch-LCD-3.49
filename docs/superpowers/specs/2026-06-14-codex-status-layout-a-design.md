# Codex 状态屏 A 方案界面重设计规格

## 目标

在保留陪伴性角色的基础上，重设计 `Examples/Arduino/12_Codex_Status` 的 LVGL 状态屏界面。目标屏幕仍是开发板横向显示的 640 x 172 画布，不引入网页运行时；网页预览仅作为设计沟通工具。

采用 A 方案：左侧角色锚点 + 中部主任务详情 + 右侧次要会话。新的界面应让用户一眼看到“当前最需要关注的 Codex 会话、它处于什么状态、下一步是什么、系统连接是否正常”，同时保留角色作为陪伴状态的视觉中心。

## 范围

本次设计覆盖以下内容：

- 重排 LVGL 屏幕布局，替换当前左列表 + 右大角色图的结构。
- 继续使用现有 `CodexStatus` / `CodexSession` 数据结构，不扩大 bridge JSON 协议。
- 继续展示状态角色图，角色图根据连接状态和最重要会话状态切换。
- 最多解析 5 个会话；屏幕上展示 1 个主会话和最多 2 个次要会话。
- 保持现有 Wi-Fi、HTTP 轮询、串口遥测、SD TTF 字体加载机制。

本次设计不覆盖以下内容：

- 不把网页部署到开发板。
- 不增加触摸分页或二级详情页。
- 不重写 Python exporter/bridge 的数据来源。
- 不新增复杂动画；角色图切换保持静态图片替换。

## 信息架构

屏幕分为三列：

1. 左侧陪伴栏，宽约 150 px。
   显示角色图、角色标题、当前整体状态、网络/刷新摘要。该区域负责维持陪伴感和整体情绪。

2. 中部主会话栏，宽约 320 px。
   显示优先级最高的会话。内容包括状态标签、标题、项目名、更新时间、进度感条、下一步提示、验证提示。该区域是信息主焦点。

3. 右侧次要会话栏，宽约 150 px。
   显示最多 2 个次要会话卡片。每个卡片包含状态标签和短标题，用于快速判断还有哪些会话需要注意。

会话优先级沿用现有排序逻辑：`blocked` 优先，其次 `active`、`complete`、`notLoaded`、未知状态。主会话取排序后的第 1 个，右侧显示第 2 和第 3 个。

## 视觉设计

整体视觉保持深色工作台风格，但减少大面积空白角色图，增加信息密度。

- 背景：`#101418`。
- 面板：`#151B20`，卡片：`#1D252C`。
- 文字：主文字 `#F7FAFC`，次级文字 `#8FA1AD` / `#C6D1D8`。
- 状态色沿用现有语义：`active` 绿色、`complete` 蓝色、`blocked` 红色、`notLoaded` 琥珀色、未知灰色。
- 圆角控制在 4 到 7 px，适配小屏幕，不使用过大的装饰卡片。
- 所有文本使用点状截断或固定高度换行，避免挤压布局。

角色图不再占据 410 x 160 的右侧大区，而是嵌入左侧陪伴栏。角色图建议显示在约 54 x 92 到 72 x 120 的框内，具体尺寸以现有图片可读性和编译后屏幕观感为准。

## 组件

### 陪伴栏

包含：

- 角色图片对象 `assistant_image`。
- 标题标签，固定为 `Codex` 或 `Codex 伙伴`。
- 整体状态标签，根据主会话或连接错误显示 `工作中`、`已完成`、`已阻塞`、`未加载`、`离线`。
- 连接摘要标签，例如 `网络已连接 · 3 秒刷新 · 5 个会话`。

当 Wi-Fi 断开、HTTP 错误或连接异常时，角色图切换到错误态，整体状态标签显示离线或服务错误。

### 主会话栏

包含：

- 主会话标题，优先显示 `session.title`，为空时显示 `未命名会话`。
- 状态标签，使用 `session.status_zh`，为空时按 `state` 映射。
- 元信息，显示 `cwd` 和 `updated_at`，为空时用 `--`。
- 进度感条，不代表真实百分比，只作为状态氛围：阻塞态约 20%，工作态约 65%，完成态 100%，未加载态约 35%，未知态约 10%。
- 下一步提示，当前 JSON 没有真实下一步字段，因此使用状态派生文案：
  - `active`：`下一步：继续处理当前任务`
  - `blocked`：`下一步：等待处理阻塞`
  - `complete`：`下一步：查看结果或收尾`
  - `notLoaded`：`下一步：加载会话后继续`
  - 未知：`下一步：等待状态更新`
- 验证提示，固定展示当前项目约束，例如 `验证：编译 + 真机日志`。

### 次要会话栏

包含 2 个固定高度卡片。每个卡片显示状态标签和标题。没有足够会话时隐藏对应卡片。

卡片不滚动，不放置长元信息，避免右侧过密。

## 数据流

现有数据流保持不变：

1. ESP32 连接 Wi-Fi。
2. 每 3 秒请求 `STATUS_URL`。
3. `parse_status_json()` 解析 `updated_at` 和最多 5 个 `sessions`。
4. `build_visible_session_order()` 按状态优先级排序。
5. `update_status_ui()` 将排序结果映射到陪伴栏、主会话栏、次要会话栏。
6. `log_status_snapshot()` 保持串口遥测输出，便于 `verify_codex_status.cmd` 判断 UI 已更新。

新增的 UI 文案必须纳入中文字体覆盖。若继续依赖嵌入式子集字体，需要扩展 `lv_font_codex_zh_16.c` 和 `lv_font_codex_zh_20.c` 的字符集；若 SD TTF 字体成功加载，嵌入字体仍作为 fallback。

## 错误处理

- Wi-Fi 连接中：主会话为空，陪伴栏显示 `连接中`，角色使用 idle 或 work 图。
- Wi-Fi 超时/断开：陪伴栏显示 `离线`，主会话区域显示网络错误摘要，角色使用 error 图。
- HTTP 错误：陪伴栏显示 `服务错误`，主会话区域显示 HTTP 状态，角色使用 error 图。
- 没有会话：主会话区域显示 `暂无会话`，次要会话卡片隐藏，角色使用 idle 图。
- 字体加载失败：继续使用嵌入式中文子集字体，同时串口输出现有 fallback 日志。

## 测试与验证

实现时按测试优先推进：

- 更新或新增静态测试，断言草图使用 A 方案中的三列区域常量、主会话对象、次要会话对象和角色左锚点。
- 保留现有图片映射测试，调整不再要求右侧 410 x 160 大角色面板。
- 更新文本布局测试，确认会话行和文本容器固定高度、不可滚动，并使用截断策略。
- 运行 `python -m unittest discover -s tests -v`。
- 固件改动完成后运行 `verify_codex_status.cmd`，要求编译、上传、串口日志和 `CODEX_STATUS ui_update` 均通过。

## 验收标准

- 开发板屏幕呈现 A 方案三列结构。
- 角色仍随状态变化，并位于左侧陪伴栏。
- 屏幕能显示比当前版本更详细的主会话信息，包括标题、状态、项目、更新时间、下一步和验证提示。
- 右侧最多展示 2 个次要会话，文本不溢出、不重叠。
- 连接错误、无会话、字体 fallback 不破坏布局。
- Python 测试通过；真机验证链路通过后才能声明固件改动完成。
